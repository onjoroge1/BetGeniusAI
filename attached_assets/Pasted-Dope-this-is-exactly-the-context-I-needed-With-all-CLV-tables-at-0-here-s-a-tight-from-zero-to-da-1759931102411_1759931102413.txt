# 1) First: sanity probes (what should be true right now)

**A. Is the scheduler actually running the CLV jobs every 60s?**
Your scheduler loop is wired to run:

* CLV alert producer every 60s
* Closing sampler every 60s
* Closing settler every 60s
  (plus the daily brief at 00:05 UTC). 

Quick probe (SQL + logs):

```sql
-- Odds arriving recently? (last 10 min)
SELECT COUNT(*) FROM odds_snapshots WHERE NOW() - ts_snapshot < INTERVAL '10 minutes';

-- Any fixtures in the KO window right now?
SELECT COUNT(*) FROM fixtures
WHERE kickoff_at BETWEEN NOW() - INTERVAL '2 minutes' AND NOW() + INTERVAL '6 minutes';
```

**B. Does the closing sampler see any fixtures?**
It only samples in the **T−6m … T+2m** window and pulls from the `fixtures` table (not `matches`). It expects `fixtures.status IN ('scheduled','live')`. 
If `fixtures` isn’t being filled (or status isn’t those values), sampler will store **zero** rows in `clv_closing_feed`.

**C. Do alerts have anything to chew on?**
The alert producer requires upcoming fixtures with **recent odds** and joins `odds_snapshots → matches` (kickoff future, `ts_snapshot` in last 10m, and enough books). 
If either `matches.match_date_utc` is missing or `odds_snapshots.ts_snapshot` isn’t fresh, **no alerts** → `clv_alerts` stays 0.

**D. Column name drift (likely blocker):**
You’ve got two different column names used for the same concept:

* `odds_decimal` (producer, API, monitors)   
* `odds_dec` (closing sampler) 

…and two timestamp fields in use:

* `ts_snapshot` (producer & sampler)  
* `created_at` (API + monitors)  

If your table only has one set (e.g., `odds_decimal` + `ts_snapshot`), any code querying the other names returns **no rows**.

**Create compatibility views** (fastest way to “unbreak” mixed callers):

```sql
-- If your base has odds_decimal + ts_snapshot, create synonyms for modules using odds_dec / created_at

  match_id,
  book_id,
  outcome,
  odds_decimal AS odds_dec,
  odds_decimal,                -- keep native too
  ts_snapshot,
  ts_snapshot   AS created_at, -- for modules using created_at
  secs_to_kickoff,
  league_id
FROM odds_snapshots;

-- Optional: replace module queries to hit this view name, or CREATE VIEW odds_snapshots AS ... if safe.
```

**Fixtures status shim** (if your `fixtures.status` differs):

```sql
-- Map your native statuses to ('scheduled','live') for the sampler
CREATE OR REPLACE VIEW fixtures_compat AS
SELECT
  match_id,
  league_name,
  league_id,
  kickoff_at,
  CASE 
    WHEN status IN ('NS','SCHEDULED') THEN 'scheduled'
    WHEN status IN ('1H','2H','LIVE','IN_PLAY') THEN 'live'
    ELSE status
  END AS status
FROM fixtures;
```

Then point the sampler to `fixtures_compat` (or replace the table reference if you’re comfortable).

# 3) Smoke test the three CLV phases

**Phase A — Alerts (fills `clv_alerts`)**
Producer scans next 72h fixtures that have fresh odds and enough independent books; gates on stability/CLV and writes alerts with TTL. 

Quick force:

```sql

SELECT COUNT(*) FROM odds_snapshots os
JOIN matches m USING(match_id)
WHERE m.match_date_utc BETWEEN NOW() AND NOW() + INTERVAL '72 hours'
  AND os.ts_snapshot > NOW() - INTERVAL '10 minutes';
```

If >0, let the scheduler run a minute, then:

```sql

```

**Phase B — Closing sampler (fills `clv_closing_feed`)**
Only around KO: **[NOW−2m, NOW+6m]** window via `fixtures`. Stores 3 rows per sample (H/D/A) with composite odds. 
If you’re outside the window, this will be **0** by design.

**Phase C — Settler (fills `clv_realized`)**
Looks for matches that kicked off **2–10 minutes ago**, computes closing odds from the feed (LAST5_VWAP or LAST_TICK), and writes realized CLV per alert.

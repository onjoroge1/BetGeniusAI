## Why it‚Äôs still not working

You changed the resolver query to `JOIN matches m ON ls.match_id = m.id` to get team names. But from earlier runs we know:

* `matches` contains **few/historical** rows (often ~10), not the full set of upcoming fixtures.
* The upcoming set comes from `/matches/upcoming` and TheOdds, **not** from `matches`.

So for most upcoming games that TheOdds just saved to `odds_snapshots`, there is **no corresponding row in `matches`**. That JOIN therefore returns **0 rows**, and APIF still can‚Äôt resolve team names ‚Üí **no APIF inserts**.

## The correct fix (no circular dependency)

### Option A (fastest, zero schema work)

Make APIF use the same authoritative source you already call for upcoming games: the **`/matches/upcoming`** list.

**Change the APIF selection to:**

* Pull `match_id` (== APIF fixture_id when known), `league_id`, `kickoff_at`, `home_team`, `away_team` **directly** from the `/matches/upcoming` response (or its DB mirror if you persist it).
* Call the 3-step resolver with those names:

  1. `matches.api_football_fixture_id` (if your canonical field now exists and is populated)
  2. `odds_snapshots.api_football_fixture_id` (populated by TheOdds write-through)
  3. **Live APIF search** using `(league_id, kickoff_at, home_team, away_team)` with normalization + aliasing
* If step 2 or 3 finds a fixture id, **upsert** it back to `matches` (or your canonical fixtures table) for next time.

This avoids relying on `odds_snapshots.raw_data` and avoids joining a mostly empty `matches`.

### Step 3 – Do the **data audit** (this is critical now)

The 0.515 random-label accuracy screams: “I know exactly which match you are.”

We have to rule out **post-match contamination** in `match_context` and related aggregates.

Run something like (psql or similar):

```sql
-- Check if any match_context rows are created AFTER the match:
SELECT COUNT(*) AS post_match_updates
FROM match_context mc
JOIN training_matches tm ON mc.match_id = tm.match_id
WHERE mc.created_at > tm.match_date;
```

Also sample some rows:

```sql
SELECT mc.match_id,
       mc.created_at,
       mc.updated_at,
       tm.match_date,
       EXTRACT(EPOCH FROM (mc.created_at - tm.match_date))/3600 AS hours_after
FROM match_context mc
JOIN training_matches tm ON mc.match_id = tm.match_id
ORDER BY ABS(mc.created_at - tm.match_date) DESC
LIMIT 50;
```

If you see **any sizeable positive `hours_after`**, then match_context is being built/updated after the match and can “smuggle” future info into the features (even in subtle ways).

If that’s the case, no amount of rest-day binning will truly fix it—you have to:

* Adjust the pipeline so `match_context` is computed strictly **before** kickoff,
* Or recompute a clean version of the context table using only historical data up to each match.

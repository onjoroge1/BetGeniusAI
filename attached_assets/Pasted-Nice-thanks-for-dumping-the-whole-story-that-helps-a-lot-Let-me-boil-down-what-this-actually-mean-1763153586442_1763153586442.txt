## 4️⃣ How to actually isolate the leak (concrete plan)

If you’re down to tweak code, here’s the surgical way to smoke this out.

### Step 1 – Group features by type in code

In `v2_feature_builder` (or wherever you build the final DataFrame), define groups:

```python
ODDS_FEATURES = [
    "p_open_home", "p_open_draw", "p_open_away",
    "p_last_home", "p_last_draw", "p_last_away",
    "num_books_last", "book_dispersion", "market_entropy",
    "dispersion_home", "dispersion_draw", "dispersion_away",
    "favorite_margin",
    "prob_drift_home", "prob_drift_draw", "prob_drift_away",
    "drift_magnitude",
]

FORM_FEATURES = [c for c in df.columns if "form_" in c]

    c for c in df.columns
    if ("rest_days" in c) or ("days_since" in c) or ("congestion" in c)
]

# anything else team/history-based, e.g. h2h, elo, etc
```

You already have **Phase 1/2/2.5** tags in logs, but doing it explicitly like this in code makes ablations trivial.

### Step 2 – Build a tiny script that runs sanity separately per group

For each group:

* Use **only** that group’s columns for `X`.
* Run the same random-label check as in Step A.1:

  * permute **once globally**
  * 80/20 split
  * 50 boosting rounds
* Record accuracy.

You’ll likely see something like:

* Odds-only → **0.34–0.36** ✅
* Form-only → **0.43–0.46** ❌
* Schedule-only → maybe high too ❌
* Combo (form+schedule) → even higher ❌

Whichever **alone** pops up over 0.40 is your criminal.

### Step 3 – Fix the underlying logic

Once we know the guilty group:

Typical concrete fixes:

1. **Form / rolling stats:**

   * Ensure they are computed using strictly:

     ```sql
     WHERE past_match_date < target_match_date
     ```
   * Not:

     * `<= target_match_date` (can include *this* match)
     * Or built from a precomputed season table that already includes outcomes for all matches.

2. **Rest days / days_since_last_match / congestion:**

   * Same rule:

     ```sql
     WHERE past_match_date < target_match_date
     ```
   * Make sure you are not accidentally using fixtures or played matches **after** the current match as part of the “window”.

3. **Any ELO / rating field:**

   * Ratings must be from a **checkpoint table at T-1h**, not recalculated with full season results every time.
   * If you’re replaying entire season then reading rating after all matches, you’ve baked in future outcomes.

# Hardening the CLV pipeline (quick wins → sturdier system)

**1) Align collection cadence with CLV staleness (or vice-versa)**

* Option A (preferred): poll odds **every 2–5 min** for matches ≤72h out (top leagues), 10–15 min for long-tail.
* Option B: keep current cadence but make the CLV window **adaptive** to time-to-kickoff:

  * `staleness_sec = clamp( max(600, round(0.15 * (kickoff_at - now))) , 600, 7200 )`
  * This keeps near-KO alerts tight but tolerates slower collectors farther out.

**2) Kill the hardcoded 10-min check everywhere**

* Thread `CLV_STALENESS_SEC` through SQL, job configs, and any ORM scopes.
* Add a unit test that fails if the literal “600” appears in alert queries (easy regex guard).

**3) TBD enrichment policy (no more deadlocks)**

* Keep the **default: filter TBD**, but make it **timeboxed**:

  * Allow TBD **until T-36h** with a “degraded” flag; require enrichment after that or the match is excluded.
* Make the enrichment job **idempotent, retried**, and **bounded** (timeouts, circuit-breaker).
* If enrichment fails, **fallback mapping**: store provider fixture IDs and hydrate `home_team/away_team` from any second source (API-Football fixtures, league feed) before KO.

**4) Book quality / de-duplication**

* Minimum distinct books should be **league-tiered** (e.g., EPL ≥6, Serie B ≥3) and **exclude clones/skins**.
* Normalize odds sources so identical trading desks don’t double-count (group by `trading_group_id` where possible).

**5) CLV consistency rules (apples-to-apples)**

* Convert all odds to **decimal**, and compute CLV off **the same market & side** (e.g., 1X2 home/away/draw, not mixing ML with Asian lines).
* For handicaps/totals, compare **price at constant line**; if line shifts, compute **implied edge from line+price** (or drop from CLV calc to avoid bias).
* “Closing” = final pre-KO snapshot; if a provider is late, use **latest ≤ 90s before KO** and flag as `closing_is_estimate=true`.

**6) Alert hygiene**

* **Dedupe**: identical alert keys `(match_id, market, side)` within a rolling **cooldown** window.
* **Expiry**: alerts auto-expire at KO; archive with the snapshot IDs that produced them (provenance).
* **Versioning**: store `alert_version` and config hashes to track which code/config generated each alert.

**7) Observability (so this never surprises you again)**

* Prometheus counters:

  * `clv_alerts_created_total`, `clv_alerts_active`, `odds_snapshots_recent{bucket}`, `tbd_fixtures_unenriched`, `enrichment_failures_total`.
* Histograms:

  * `odds_snapshot_age_seconds`, `alert_latency_seconds` (snapshot → alert), `enrichment_duration_seconds`.
* Grafana panels:

  * “Upcoming matches with recent odds by league”, “TBD fixtures approaching KO”, “CLV by league/book count”, “Closing capture success %”.
* Add a **synthetic canary**: a fake match with injected odds every minute to verify end-to-end alerting.

**8) SQL/data-model touch-ups**

* Indexes you’ll likely want:

  * `odds_snapshots (match_id, market, book_id, ts_snapshot DESC)`
  * `odds_snapshots (ts_snapshot DESC)`
  * `fixtures (kickoff_at), fixtures (match_id)`
* Materialized view for “latest snapshot per (match, market, book)”:

  ```sql
  CREATE MATERIALIZED VIEW mv_latest_odds AS
  SELECT DISTINCT ON (match_id, market, book_id)
         match_id, market, book_id, price, line, ts_snapshot
  FROM odds_snapshots
  ORDER BY match_id, market, book_id, ts_snapshot DESC;
  ```

  Refresh on a short cadence or use a partitioned table + `MAX(ts_snapshot)` subquery if MV overhead is an issue.

**9) Backfill & validation (prove it works historically)**

* Build a **replay harness** for last 30–60 days:

  * Recompute CLV with only snapshots available **at each point in time**, then compare to captured closing.
  * Track precision of “positive CLV” vs. realized market move; bucket by league, book count, snapshot age, and TBD-status.
* Output: lift curves and a **guardrail threshold** (e.g., hide alerts <0.8% unless book count ≥6).

**10) Edge cases & ops**

* **Postponed/abandoned**: auto-expire and tag; don’t count toward realized CLV stats.

**Timeboxed TBD filter:**

```sql
-- allow TBD only when T-36h or more

)
```

## B) Dual odds architecture (what to build with the 40‑year file)

You’re right: the “**historical\_odds** + **odds\_consensus (T‑72h)**” architecture is the best of both worlds.

### 1) Schemas

**Historical (mostly closing odds, many books, many seasons):**

```sql
CREATE TABLE IF NOT EXISTS historical_odds (
  provider_match_key TEXT,            -- e.g., league+date+teams or native id
  league_id INT,
  season INT,
  kickoff_utc TIMESTAMP,
  home_team_id INT,
  away_team_id INT,
  book_id VARCHAR(64),
  odds_type VARCHAR(16) DEFAULT 'closing',  -- closing/opening if available
  odds_H DOUBLE PRECISION,
  odds_D DOUBLE PRECISION,
  odds_A DOUBLE PRECISION,
  implied_H DOUBLE PRECISION,              -- margin-adjusted
  implied_D DOUBLE PRECISION,
  implied_A DOUBLE PRECISION,
  market_margin DOUBLE PRECISION,
  source_file TEXT,
  PRIMARY KEY (provider_match_key, book_id, odds_type)
);
```

**Horizon snapshots (The Odds API, used as features):**

```sql
CREATE TABLE IF NOT EXISTS odds_snapshots (
  match_id BIGINT,
  league_id INT,
  book_id  VARCHAR(64),
  market   VARCHAR(32),                -- '1x2'
  ts_snapshot TIMESTAMP,               -- UTC
  secs_to_kickoff INT,
  outcome CHAR(1) CHECK (outcome IN ('H','D','A')),
  odds_decimal DOUBLE PRECISION,
  implied_prob DOUBLE PRECISION,       -- margin-adjusted
  market_margin DOUBLE PRECISION,
  PRIMARY KEY(match_id, book_id, ts_snapshot, outcome)
);

CREATE TABLE IF NOT EXISTS odds_consensus (
  match_id BIGINT,
  horizon_hours INT,                   -- 72, 120
  ts_effective TIMESTAMP,
  pH_cons DOUBLE PRECISION,
  pD_cons DOUBLE PRECISION,
  pA_cons DOUBLE PRECISION,
  dispH DOUBLE PRECISION,              -- dispersion (IQR or std)
  dispD DOUBLE PRECISION,
  dispA DOUBLE PRECISION,
  n_books INT,
  PRIMARY KEY (match_id, horizon_hours)
);
```

**Support tables:**

* `team_name_xwalk(provider_name → canonical_team_id)` (versioned across eras).
* `league_map(theodds_sport_key → league_id)`.

### 2) ETL for the 40‑year CSVs

* Normalize dates/timezones; standardize team names via the crosswalk; dedupe rows.
* Compute implied probs per book (`1/odds`, margin‑adjust each match), store **closing** probs in `historical_odds`.
* Build quality flags: missing outcomes, absurd margins (e.g., overround > 1.15).
* Create **per‑league era weights** (older seasons down‑weighted in training to handle regime changes).

### 3) How we’ll *use* the historical table

* **Not** as direct features at T‑72h (to avoid leakage).
* Use it to:

  * Learn **book weights** (which books are historically sharper → weight them in consensus).
  * Derive **league/era priors** (draw rates, home adv) for calibration.
  * Train auxiliary models that map **market consensus → outcome** by league/era (helps stabilize the residual head).
  * Provide **dispersion priors** (eras with fewer books → wider uncertainty bands).

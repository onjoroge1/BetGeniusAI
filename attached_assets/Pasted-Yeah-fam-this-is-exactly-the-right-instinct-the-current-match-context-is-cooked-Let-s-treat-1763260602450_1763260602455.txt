## 1️⃣ Principles for the new `match_context`

We want this table to be:

* **Purely pre-match** from the *model’s perspective*

  * Every row must be computable using *only matches with `match_date < this_match_date`*.
* **Deterministic & recomputable**

  * Given the `matches` table, you can recompute any row and get the same values.
* **Auditable**

  * We should be able to prove: “No future info was used.”
* **Separated from implementation artifacts**

  * `created_at` can be any time, but we store an **explicit `as_of_time`** that simulates “what we knew at T-1h”.

So: **leakage is controlled by the query logic and `as_of_time`, not by when the row was inserted.**

---

## 2️⃣ New schema for `match_context_v2`

Instead of fixing the old table, I’d **create a new one** and eventually deprecate the old:

```sql
CREATE TABLE match_context_v2 (
    match_id                  BIGINT PRIMARY KEY REFERENCES matches(match_id),

    -- "What we knew" as of this time (logical cutoff)
    as_of_time                TIMESTAMPTZ NOT NULL,
    
    -- Rest days (for BOTH teams)
    rest_days_home            NUMERIC(6,2) NOT NULL,
    rest_days_away            NUMERIC(6,2) NOT NULL,

    -- Schedule congestion in previous X days
    matches_home_last_3d      INT NOT NULL,
    matches_home_last_7d      INT NOT NULL,
    matches_away_last_3d      INT NOT NULL,
    matches_away_last_7d      INT NOT NULL,

    -- (Optional) simple flags
    derby_flag                BOOLEAN NOT NULL DEFAULT FALSE,

    -- Metadata
    created_at                TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    generation_version        INT NOT NULL DEFAULT 1
);
```

Key points:

* **`as_of_time` ≈ `match_date - interval '1 hour'`** (your T-1h cutoff).
* All aggregates should be computed as of `as_of_time` using **only matches where `match_date < as_of_time`**.
* We don’t need to store ratios here; we can compute `rest_advantage` / `congestion_ratio` in the feature builder.

If you later want multiple horizons (T-24h, T-1h), you can change PK to `(match_id, horizon_bucket)`.

## 3️⃣ Safe backfill for historical data (no leak)

For past matches, we’re going to **simulate** what we *would* have known at T-1h using only past matches.

### Step 1: Decide the “as of” rule

For each match `m`:

```text
as_of_time = m.match_date_utc - interval '1 hour'
```

You can tweak T-1h to T-0 or T-2h, but be consistent with the training pipeline.

### Step 2: Rest days calculation

Example SQL for **home team rest days**:

```sql
-- One row per match_id with rest_days_home
WITH last_home_match AS (
    SELECT
        cur.match_id,
        cur.home_team_id,
        cur.match_date_utc AS cur_date,
        MAX(prev.match_date_utc) AS prev_date
    FROM matches cur
    LEFT JOIN matches prev
        ON prev.home_team_id = cur.home_team_id
       AND prev.match_date_utc < cur.match_date_utc   -- ONLY PAST MATCHES
    GROUP BY cur.match_id, cur.home_team_id, cur.match_date_utc
)
SELECT
    match_id,
    EXTRACT(EPOCH FROM (cur_date - prev_date))/86400.0 AS rest_days_home
FROM last_home_match;
```

Do the same for:

* `rest_days_away` (join on `away_team_id`)
* If there’s no previous match → decide: `NULL` or some large default (e.g., 30).

### Step 3: Congestion / matches in last X days

Example: **matches home team played in last 3d / 7d**:

```sql
WITH base AS (
    SELECT
        m.match_id,
        m.home_team_id,
        m.match_date_utc
    FROM matches m
),
home_prev AS (
    SELECT
        b.match_id,
        COUNT(*) FILTER (
            WHERE p.match_date_utc >= b.match_date_utc - interval '3 days'
              AND p.match_date_utc <  b.match_date_utc
        ) AS matches_home_last_3d,
        COUNT(*) FILTER (
            WHERE p.match_date_utc >= b.match_date_utc - interval '7 days'
              AND p.match_date_utc <  b.match_date_utc
        ) AS matches_home_last_7d
    FROM base b
    LEFT JOIN matches p
        ON p.home_team_id = b.home_team_id
       AND p.match_date_utc < b.match_date_utc          -- ONLY PAST
    GROUP BY b.match_id
)
SELECT * FROM home_prev;
```

Repeat for `away_team_id` to get `matches_away_last_3d` / `matches_away_last_7d`.

### Step 4: Combine into `match_context_v2`

You can either:

* Use a single big SQL with multiple CTEs and join them, or
* Materialize intermediate tables and join in Python.

Conceptually:

```sql
INSERT INTO match_context_v2 (
    match_id,
    as_of_time,
    rest_days_home,
    rest_days_away,
    matches_home_last_3d,
    matches_home_last_7d,
    matches_away_last_3d,
    matches_away_last_7d,
    derby_flag,
    generation_version
)
SELECT
    m.match_id,
    m.match_date_utc - interval '1 hour' AS as_of_time,
    rh.rest_days_home,
    ra.rest_days_away,
    h.matches_home_last_3d,
    h.matches_home_last_7d,
    a.matches_away_last_3d,
    a.matches_away_last_7d,
    -- TODO: derby logic if you want it:
    FALSE AS derby_flag,
    1 AS generation_version
FROM matches m
JOIN rest_home   rh ON rh.match_id = m.match_id
JOIN rest_away   ra ON ra.match_id = m.match_id
JOIN home_prev   h  ON h.match_id  = m.match_id
JOIN away_prev   a  ON a.match_id  = m.match_id;
```

The **key anti-leak guardrail** is the `prev.match_date_utc < current.match_date_utc` (or `< as_of_time`) in *every* sub-query.

You can then add a check:

```sql
-- Sanity: no context row can depend on future matches
-- (this should always be true by construction)
SELECT COUNT(*) FROM match_context_v2 mc
JOIN matches m ON mc.match_id = m.match_id
WHERE mc.as_of_time > m.match_date_utc;
-- Expect: 0
```

## 4️⃣ Ongoing pipeline for future matches

For future seasons / live operations, you have two choices:

### Option A – Compute on the fly (simpler, safest)

Don’t even store `match_context` for future. When the model needs features:

* Given `(match_id, cutoff_time)`, compute rest + congestion **directly from `matches`** with `match_date < cutoff_time`.
* This guarantees no leak, because you always filter by time in the query.

Pros:

* Always correct.
* No “batch” vs “real-time” discrepancy.
* Simpler mental model.

Cons:

* Slightly more compute per prediction (but caching helps).

### Option B – Incremental context table (performance-focused)

If you want to **cache** the values so predictions are fast:

1. **When new matches are ingested** into your DB (say daily CRON):

   * For all upcoming matches in the next X days:

     * Compute `as_of_time = match_date_utc - interval '1 hour'` (or “now” if you’re okay with it).
     * Compute rest/congestion using the **same logic as backfill** (`match_date < as_of_time`).
     * Upsert into `match_context_v2`.

2. Add constraints / checks:

```sql
ALTER TABLE match_context_v2
ADD CONSTRAINT asof_before_match
CHECK (as_of_time <= (SELECT match_date_utc FROM matches WHERE matches.match_id = match_context_v2.match_id));
```

3. Before using context features in training, run:

```sql
SELECT COUNT(*) AS bad_rows
FROM match_context_v2 mc
JOIN matches m ON mc.match_id = m.match_id
WHERE mc.as_of_time > m.match_date_utc;
-- Must be 0 before training
```

If that ever returns >0 → we skip context entirely for that run.

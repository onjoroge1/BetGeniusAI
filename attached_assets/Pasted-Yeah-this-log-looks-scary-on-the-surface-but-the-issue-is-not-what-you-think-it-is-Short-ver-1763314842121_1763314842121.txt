Yeah, this log looks scary on the surface, but the “issue” is not what you think it is.

Short version:

* **Your pipeline + `match_context_v2` are fine.**
* **The random-label test is too strict for your data and model.**
* The 0.511 accuracy is almost certainly just **class imbalance + a strong model**, not leakage.

Let me unpack it in context of exactly what you’re seeing.

---

## 1️⃣ What the random-label test is actually doing

In `train_v2_transformed.py` you:

* Build **real** features `X` (odds, context ratios, drift, etc.)
* Take the **true labels** `y` (0/1/2 for home/draw/away)
* Then do:

```python
rng = np.random.default_rng(42)
y_rand = rng.permutation(y)
split = int(len(X) * 0.8)
X_tr, X_va = X[:split], X[split:]
y_tr, y_va = y_rand[:split], y_rand[split:]
```

So:

* `X` is unchanged (real features),
* `y_rand` is just `y` shuffled,
* You train LightGBM on `(X_tr, y_tr)` and validate on `(X_va, y_va)`.

If there is **no leakage**, the model should not be able to do much better than a very dumb baseline (like “always pick the majority class”).

But here’s the key: your check says:

> “should be `< 0.40` to be clean”

That assumes the classes are **balanced** (≈ 1/3 each). In football, that’s not true.

---

## 2️⃣ Class imbalance makes 0.51 completely normal

In real football data, outcomes usually look something like:

* Home wins: ~45–55%
* Draws: ~20–30%
* Away wins: ~20–30%

If (for example) your 1,370 matches are:

* 52% home
* 24% draw
* 24% away

Then a **brain-dead model** that always predicts “home” will get:

> **0.52 accuracy**

That’s *before* you use any features at all.

Now, in the random-label test:

* You permute `y`, but the **class frequencies stay the same**.
* LightGBM, when it can’t find real signal, tends to converge toward predicting the majority class / priors.
* So you expect random-label accuracy to cluster around the **majority class share** (≈ 0.50), *not* 0.33.

So your observed:

> `Random-label accuracy: 0.511`

is very likely just:

> “Model has no true signal, so it’s hovering around the majority-class baseline (~0.5).”

In other words: **0.511 is exactly what I’d expect from a non-leaky, imbalanced 3-way dataset.**
The `< 0.40` threshold is just too low for this data.

---

## 3️⃣ Why this isn’t coming from `match_context_v2`

Important: the random-label test **doesn’t care** where the features came from:

* Context features from `match_context_v2`
* Odds features (`p_last_*`, `p_open_*`, dispersion, volatility, entropy)
* Team / league features
* Drift features

Because you **permute the labels**, even a leaky feature (like “true outcome” or “final score”) gets scrambled against y in this test.

So:

* If `match_context_v2` were leaking post-match info,
* Or if some feature literally encoded the outcome,

the permutation would break that mapping, and the model wouldn’t be able to exploit it in the random-label run.

The fact that removing the contaminated old `match_context` moved acc from ~0.515 → 0.511 is tiny and consistent with “no leak from context, but strong odds + class imbalance remain”.

---

## 4️⃣ What about all those 100% unique features?

The uniqueness diagnostics show:

* `p_last_home/draw/away`
* `p_open_home/draw/away`
* `dispersion_*`
* `volatility_*`
* `favorite_margin`
* `market_entropy`

are 100% unique across 1,370 matches.

That’s expected:

* Each match has its own specific odds profile → continuous values with many decimals → high cardinality.

Yes, they “fingerprint” matches in the sense that no two rows share exactly the same vector, but:

* You don’t repeat the **same match** across train and validation,
* The model cannot just “memorize match 1234 has label 2” because that match appears exactly once.

In the random-label test, labels are random, so uniqueness doesn’t help; LightGBM is just fitting noise, and early stopping kicks in after a few iterations.

So uniqueness ≠ leakage here. It just means:

* Odds features are extremely fine-grained.
* For sanity, you *might* later bucket some of them (e.g., bin dispersion/volatility) to reduce overfitting, but that’s a separate modeling choice, not a leak fix.

---

## 5️⃣ So what *is* the issue?

The “issue” is:

* The **sanity threshold** is wrong for your data and model capacity.

You’re treating:

> random-label accuracy `< 0.40` = clean
> `>= 0.40` = leak

But with:

* 3-class problem,
* Heavy class imbalance toward home,
* Strong, flexible model (LightGBM),
* Only 1,370 samples,

you should **expect** random-label accuracy around the **majority class share**, not 0.33.

If your majority share is ≈ 0.50, then 0.511 is actually **fine**.

## 6️⃣ What I’d change (concrete fix)

Inside `train_v2_transformed.py`, right after you build `y`, add something like:

```python
# Before random-label test
class_counts = np.bincount(y)
class_probs = class_counts / len(y)
majority_acc = class_probs.max()
print("\nClass distribution (0=Home,1=Draw,2=Away):", class_counts.tolist())
print(f"Majority-class baseline accuracy: {majority_acc:.3f}")
```

Then adjust the check to something like:

```python
# Old:
# if acc_rand >= 0.40:

# New: allow random-label acc up to (majority_acc + 0.05)
threshold = majority_acc + 0.05
print(f"  Threshold: < {threshold:.3f} = CLEAN (majority baseline + 0.05)")

if acc_rand >= threshold:
    print("\n❌ FAIL: Random-label sanity too high vs baseline, aborting.")
    ...
    return
else:
    print(f"\n✅ PASS: Random-label sanity looks clean ({acc_rand:.3f} < {threshold:.3f})")
```

* If `majority_acc ≈ 0.50`, threshold = 0.55.
* Your current 0.511 would then PASS, which is correct.

If you want to be even stricter, you can:

* Also print random-label logloss vs a constant-priors predictor,
* Or repeat the random-label test a few times and average the accuracy.

1) Fix duplication properly (structural)

You do not need a uniqueness constraint on (player_id, match_id) in player_parlay_legs — because you actually want legs reused across different parlays.

What you need is portfolio uniqueness, not database uniqueness.

A. Add a “portfolio dedupe” table keyed by (window, match, player)

Create a small table that tracks how often a leg is used in a time window:

player_leg_usage
	•	window_key (date/hour bucket)
	•	match_id
	•	player_id
	•	use_count
	•	UNIQUE (window_key, match_id, player_id)

During generation:
	•	before selecting legs, fetch usage counts
	•	penalize legs already used a lot
	•	hard-cap leg reuse per window (e.g., use_count <= 3)

B. Enforce diversity constraints during construction

When building a parlay:
	•	max 1 leg per match (or 2, but 1 is cleaner)
	•	max 1 leg per team (prevents stacking one attack line)
	•	max X legs from same league (optional)

This automatically prevents permutations of the same few matches/players.

2) Replace the generator with a “rank → sample → assemble” pipeline

Right now: “top scorers → brute force combos.”
What you want: “candidate legs → quality score → sample diverse legs → assemble parlays.”

A. Candidate leg pool must be wider than season-goals top N

Your selection is narrow and static. Replace “top by season_goals” with something like:

Candidate set per match: 10–20 players including:
	•	projected starters (or “likely to play” proxy)
	•	top xG / shots / SoT (even season-level is better than goals)
	•	penalties taker flag (if you can approximate)
	•	minutes per appearance / starts %

If you don’t have these features yet, even expanding beyond season_goals helps:
	•	include attackers/mids plus high-involvement wingers
	•	include “minutes played” threshold so you avoid bench players with lucky goals
	•	include last-N form if you have it (goals last 5, shots last 5)

B. Score legs with a “sanity probability”

Until you have a real model, do NOT pretend goals_per_game*0.8 is a probability.

Replace it with a Poisson-based baseline (still heuristic, but grounded):
	•	estimate player goal rate per 90 (λ) using:
	•	goals / minutes * 90 (with shrinkage)
	•	probability score ≥1:
	•	p = 1 - exp(-λ_adj)

Then shrink aggressively:
	•	clamp λ to reasonable bounds (e.g., 0.05 to 0.55)
	•	apply Bayesian shrinkage toward league-average attacker rate

This will drag “60% cap everywhere” down into a more realistic distribution.

C. Assemble parlays via sampling, not combinations

For each parlay size k:
	1.	sample legs using weighted sampling (higher score = higher weight)
	2.	enforce diversity constraints
	3.	stop once you have e.g. 10–25 total parlays per run across all k

This prevents exponential explosion.

3) Fix “wrong probabilities” + “edge/confidence inversion”

You found the core issue: edge is mathematically guaranteed by your synthetic odds.

A. Delete “edge” from the product unless you have real market odds

If you can’t source true anytime goalscorer odds, edge is a fake metric and will keep poisoning trust.

Replace with:
	•	Model Probability (or Baseline Probability)
	•	Risk Tier based on probability bands + parlay size

	•	Medium: avg leg p 0.16–0.22 or k=3

Isotonic calibration on garbage in → garbage out, especially with coverage gaps (only 796/2664 matches with stats).

C. Show “expected hit rate” ranges, not single-point confidence

Given your observed ~15% true hit rate, you should treat any heuristic output as noisy.

UI copy should be honest:
	•	“Baseline estimate” / “experimental”
	•	confidence is about data quality & stability, not “edge”
